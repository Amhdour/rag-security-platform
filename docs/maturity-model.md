# Integration Maturity Model

This model describes incremental maturity for the Onyx + Starter Kit + integration-adapter workspace.

## Stage 0 — Blueprint

**Definition**
- Architecture and boundaries are documented.
- Runtime/governance/adapter separation is explicit.

**Entry criteria**
- High-level integration architecture documented.
- Unconfirmed runtime hooks explicitly marked.

**Evidence examples**
- `docs/integration-blueprint.md`

---

## Stage 1 — Adapter Scaffold

**Definition**
- Adapter package exists with schemas, mappers/translators, artifact writers, and tests.

**Entry criteria**
- Adapter can generate starter-kit-shaped artifacts from sample payloads.
- Schema and malformed-input tests exist.

**Evidence examples**
- `integration-adapter/integration_adapter/schemas.py`
- `integration-adapter/integration_adapter/mappers.py`
- `integration-adapter/integration_adapter/artifact_output.py`
- `integration-adapter/tests/`

---

## Stage 2 — Live Inventory Export

**Definition**
- Adapter inventory exporters (connectors/tools/MCP/evals metadata) are wired to confirmed runtime sources.

**Entry criteria**
- Exporters read real runtime-backed data paths for inventory domains.
- Compatibility/provenance docs pin upstream commits.

**Exit criteria**
- Inventory artifact generation is reproducible from live runtime state.
- Any unresolved fields are explicitly tagged as unconfirmed.

---

## Stage 3 — Runtime Audit Export

**Definition**
- Adapter exports normalized runtime lifecycle/policy/retrieval/tool/MCP events from confirmed runtime hooks.

**Entry criteria**
- Event vocabulary alignment is tested end-to-end.
- Fallback/deny/error semantics are preserved in normalized output.

**Exit criteria**
- At least one repeatable event export flow produces valid `audit.jsonl` + replay artifacts.

---

## Stage 4 — Launch-Gate Automation

**Definition**
- Generated adapter artifacts automatically feed launch-gate evaluation workflows.

**Entry criteria**
- Eval summaries + launch-gate reports are produced by scripted flow.
- Failure modes (malformed/missing evidence) are covered by tests.

**Exit criteria**
- Deterministic go/no-go report generation tied to artifact evidence.

---

## Stage 5 — Governed Demo Environment

**Definition**
- A fully scripted, reproducible demo environment exists with clear runtime-vs-demo separation and read-only governance UX.

**Entry criteria**
- Demo artifacts are generated by script.
- Dashboard consumes demo artifact root in read-only mode.
- Provenance and compatibility docs are up to date.

**Exit criteria**
- New contributors can run documented demo flow and inspect evidence/launch status without modifying runtime state.

---
