# Integration Maturity Model

This model describes incremental maturity for the Onyx + Starter Kit + integration-adapter workspace.

## Stage 0 — Blueprint

**Definition**
- **Implemented:** Architecture and boundaries are documented.
- **Implemented:** Runtime/governance/adapter separation is explicit.

**Entry criteria**
- High-level integration architecture documented.
- Unconfirmed runtime hooks explicitly marked.

**Evidence examples**
- `docs/integration-blueprint.md`

---

## Stage 1 — Adapter Scaffold

**Definition**
- **Implemented:** Adapter package exists with schemas, mappers/translators, artifact writers, and tests.

**Entry criteria**
- **Implemented:** Adapter can generate starter-kit-shaped artifacts from sample payloads.
- **Implemented:** Schema and malformed-input tests exist.

**Evidence examples**
- `integration-adapter/integration_adapter/schemas.py`
- `integration-adapter/integration_adapter/mappers.py`
- `integration-adapter/integration_adapter/artifact_output.py`
- `integration-adapter/tests/`

---

## Stage 2 — Live Inventory Export

**Definition**
- **Planned:** Adapter inventory exporters (connectors/tools/MCP/evals metadata) are wired to fully confirmed runtime sources.

**Entry criteria**
- **Partially Implemented:** Exporters read real runtime-backed data paths for inventory domains when optional runtime hooks are available.
- **Implemented:** Compatibility/provenance docs pin upstream commits.

**Exit criteria**
- **Partially Implemented:** Inventory artifact generation is reproducible in demo/file-backed mode; live runtime reproducibility remains environment-dependent.
- **Implemented:** Any unresolved fields are explicitly tagged as unconfirmed.

---

## Stage 3 — Runtime Audit Export

**Definition**
- **Partially Implemented:** Adapter exports normalized runtime lifecycle/policy/retrieval/tool/MCP events from file-backed and optional runtime hooks.

**Entry criteria**
- **Implemented:** Event vocabulary alignment is tested end-to-end at artifact level.
- **Implemented:** Fallback/deny/error semantics are preserved in normalized output.

**Exit criteria**
- **Implemented:** At least one repeatable event export flow produces valid `audit.jsonl` + replay artifacts.

---

## Stage 4 — Launch-Gate Automation

**Definition**
- **Implemented:** Generated adapter artifacts automatically feed launch-gate evaluation workflows.

**Entry criteria**
- **Implemented:** Eval summaries + launch-gate reports are produced by scripted flow.
- **Implemented:** Failure modes (malformed/missing evidence) are covered by tests.

**Exit criteria**
- **Implemented:** Deterministic go/no-go report generation tied to artifact evidence.

---

## Stage 5 — Governed Demo Environment

**Definition**
- **Partially Implemented:** A scripted demo environment exists with runtime-vs-demo separation and read-only governance UX; live-runtime parity remains unconfirmed.

**Entry criteria**
- **Implemented:** Demo artifacts are generated by script.
- **Implemented:** Dashboard consumes demo artifact root in read-only mode.
- **Partially Implemented:** Provenance and compatibility docs are maintained, but may need updates as upstream evolves.

**Exit criteria**
- **Implemented:** New contributors can run documented demo flow and inspect evidence/launch status without modifying runtime state.

---

  - Some runtime exporter hook sources remain explicitly unconfirmed.
